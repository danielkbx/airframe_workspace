# Project Memory

## Goal

Explore whether it is possible and worthwhile to build Airframe, a native iOS app, potentially a Universal iOS/macOS app, that provides Betaflight Blackbox log viewing similar to the official Blackbox Explorer.

## Durable Decisions

- Multi-motor Craft layout (implemented 2026-07-22): the Graph Craft inspector supports 2 through 8 motors with placement-driven rendering, known Betaflight templates, coaxial depth levels, and radial/custom fallback geometry. The existing bounded `craftTimeline` pass also streams a maximum of one mixer-estimator sample per 5 ms from motor output, optional throttle setpoint, and available P/I/D/F/S axis sums; it requires 200 accepted samples and two excited axes, solves per-motor centered regression with an internal pivoted Gaussian solver, checks both yaw polarities, and reports the specified R²/template/coverage confidence (>= 0.75 confident). Only actually excited axes participate in template matching. A 0.72 template-similarity guard prevents weak named matches from replacing coefficient-derived custom geometry. Four-motor automatic layout detection conservatively prefers Quad-X unless a special layout beats it clearly; local `btfl_007.bbl` now resolves to `quadX` instead of a marginal V-Tail false positive. Manual template overrides are sparse per-segment document state, route to active document/reference stores, sync through existing iCloud KVS state, clear with file reset, and remain outside graph presets. The layout menu is an icon-only canvas overlay; all names/status strings are localized through `AirframeCaptions`. Real-log validation identifies local `Logs/X8/LOG00027.BFL` as `octoX8` with confidence 0.966, 5,385 accepted samples, and all three control axes excited.

- Motor-count support decision (2026-07-22): data-driven views and the Craft inspector support logs with 2 through 8 motors. Graph defaults, Table defaults, Spectrum motor groups, derived motor aggregates, Timeline consumers, Craft layout/rendering, and motor-anomaly analysis use every recorded motor in that supported range. Eight-motor regression coverage verifies `motor[7]` in graph/table selection, Spectrum resolution, aggregate calculations, anomaly detection, and Craft geometry.

- Graph frame-gap rendering (implemented 2026-07-22): `BlackboxReader` records full-resolution main-frame timing gaps during `DecodedLog.scan()` before overview decimation, comparing consecutive valid main-frame `loopIteration` values and carrying `ReaderMainFrameGap` through `ReaderScanOverview`. Graph and Timeline render those gaps as real timeline discontinuities: traces/area paths break across the gap, an orange span marks the missing interval, and the main Graph shows a localized duration chip with missing-frame detail when exact loop math is available. Duplicate `invalidFrameJump` issue markers inside a known gap are suppressed without suppressing other issue kinds at the same time.

- App appearance decision (2026-07-22): Airframe forces Dark Mode at the SwiftUI scene root regardless of the user's system appearance. The iOS `WindowGroup`, macOS document windows, and macOS Settings scene all apply `.preferredColorScheme(.dark)`.

- Motor anomaly detection (implemented 2026-07-22): user-facing wording is `Motor Anomaly`; a desync is only one possible cause. `BlackboxAnalysisWorkspace.motorAnomalies(using:)` finds a per-motor interval only after that motor previously recorded useful eRPM, its normalized motor output is at least 15%, and logged eRPM (`eRPM / 100`) remains <= 1 for at least 250 ms. It uses an exact indexed `mainFrameFields` read, never the decimated scan overview. Missing fields, never-valid telemetry, low output, and short interruptions yield no result; stale non-zero eRPM remains an intentional false-negative limitation. `LogTimeline.Model` retains cached anomalies; Timeline merges only background ranges while Graph retains individual red start/end markers. Table, Spectrum, persistence, settings, and root-cause classification remain out of scope. Synthetic boundary tests are in place; an approved anonymized real log will be an end-to-end fixture later.

- Sidebar dual-selection decision (2026-07-21): the document sidebar uses two vertically stacked native SwiftUI sidebar `List`s when a log workspace is available: File/Logs/References receive 70% of the height and Presets receive the remaining 30%, separated by a `Divider` and fixed 8-point layout padding above the preset list; without a workspace the log list uses the full height. The gap must remain outside the preset list's scroll content so selection and scrolling cannot move it. Each list has its own single-selection binding, so the active log and applied preset remain selected simultaneously without a multi-selection `Set`, pending-selection mirror, custom selection background, or preset checkbox. Both lists scroll independently. All selectable log, reference-log, and preset rows are the same raw tagged `SidebarItemRow`; log rows do not need `NavigationLink` because `selectedLogID` already drives the detail and column presentation. The shared row combines its selection value with `EnvironmentValues.isFocused`: focused selections use `PrimaryOnAccent`/`SecondaryOnAccent`, while inactive gray selections use normal foregrounds. Craft names appear once on their file row, not on every child log row, and the main file row has no byte-count label.
- Graph and Table playback decision (2026-07-20): Graph and Table modes have window-scoped media-style Play/Pause transport over the shared current-position time. Playback runs from the current position to the active timeline Out point; Play at Out restarts at In; leaving both playback-capable modes or changing logs pauses. Table follows the advancing position through its existing center-locked scroll coordination. Speed is global and persisted under `airframe.playback.speedStep` in integer 0.1x steps from 0.3x through 3.0x, default 1.0x, and is not part of Reset File Settings. Position advances from a monotonic time anchor rather than accumulated timer increments so a future video player can share the same master timeline without drift.
- Graph and Table playback toolbar placement (2026-07-20): Graph and Table use the same reusable content-plus-`LogTimelineRegion` container and therefore the exact same Timeline and custom toolbar on both platforms. Playback is centered at 75% of content width between the timestamp and right-side Auto/In/Out actions. The Table timestamp stays at 50%, exactly under its center-locked cursor; Graph continues to align it to the graph cursor fraction.
- Playback control polish (2026-07-20): the Timeline transport is a 20 pt two-segment capsule: Play/Pause on the left and speed controls plus the rate readout in one right segment. The slider uses `.controlSize(.mini)`. While playing, the complete left segment through the vertical divider has the accent background. macOS exposes a top-level Playback menu through the existing focused `LogViewCommandState` capabilities/actions mechanism; Space toggles playback in Graph and Table, with Slower, Faster, and Normal Speed commands alongside it.
- Playback timestamp integration (2026-07-20): the Timeline transport is a three-segment capsule ordered Play/Pause, timestamp, then speed slider/icons/rate. A custom `TimestampAnchoredToolbarLayout` reads the timestamp segment's explicit horizontal alignment guide and places that guide at the cursor fraction, rather than centering the asymmetric whole capsule. The timestamp therefore remains exactly under the Graph cursor or Table's fixed center line while the surrounding controls become one visual unit.
- Playback initial placement (2026-07-21): Graph does not render the integrated Playback transport until it has published a valid `graphVisibleRange`; there is no provisional control position while graph layout/data is unavailable. `TimestampAnchoredToolbarLayout` receives the dimensionless cursor fraction and converts it to X using its own current layout bounds; never precompute an absolute X from an outer `GeometryReader`, whose initial width can be zero and remain stale until another state change. - Main-stream resync doctrine (2026-07-13): a rejected `I` frame becomes a tentative absolute anchor. Airframe decodes and buffers it plus following main frames, publishing the buffer only after two consecutive `P` frames satisfy normal monotonic jump checks. An out-of-window `I` may provide untrusted predictor state while the parser searches for a later local anchor, but neither it nor its `P` frames are exposed or allowed to extend the accepted timeline. A tentative anchor must remain within six normal time-jump windows of the last committed frame; typed `loggingResume` remains the only legitimate discontinuity. On failure, committed history is restored; `P` frames never anchor by themselves and unknown events never seed recovery. On the private 8.0 MiB Betaflight 4.5.2 Maya log, this recovers at byte `1,173,982` / `23.035302 s` and ends log 1 at `60.541099 s` (Upstream: `00:08–01:00`), rejecting the false ~`693 s` continuation. Log 2 is independently `104.486025–127.427661 s` (Upstream: `01:44–02:07`). Defects are user-visible: `ReaderLogIndex.nearestMainFrameTime(atOffset:)` maps issue offsets to times, timeline renders issue markers (orange `Style.issueLine`) next to event markers with shared cap and same-time dedupe.
- One-pass scan doctrine (user directive 2026-07-13, implemented in commits `3b61d63`, `046ac33`, `768ef17`, `dec5496`, `430969b`): the open scan is THE data-acquisition step and must collect everything whole-flight consumers need in one decode pass. `DecodedLogScan.overview: ReaderScanOverview` carries decimated main-frame samples (FULL field layout per sample, adaptive stride halving, budget `Configuration.maxScanOverviewSampleCount` = 1024, final count in [budget, 2x budget)) plus all `ReaderEventRecord`s (truncating at `maxEventRecordCount` with a flag instead of throwing). New series (gyro, RPM, ...) are added as projections in `BlackboxAnalysisWorkspace.overviewPoints(for:in:)` — never as another decode pass. Full-resolution zoomed views remain range queries. `summarize()`/`makeIndex()` skip overview collection and are unchanged. Concurrency doctrine: one frame stream is sequential; independent logs of a document scan in parallel (task group capped by `maxConcurrentLogTasks`); projections over the immutable Sendable overview are parallelizable by construction. Release numbers (5.7 MB log): scan 0.77 s -> 0.54 s (per-frame dictionary removal) -> 0.57 s with overview (~5% overhead); timeline data 1.2 s (viewport+events) -> 5.6 ms (projection).
- Concurrency rule (user directive 2026-07-12): data preparation must always run off the main actor, and high-frequency progress callbacks must be throttled before they reach main-actor state. Heavy worker functions must never live as members of SwiftUI View types: `View` is a MainActor protocol, its isolation propagates to all members, and detached tasks calling them synchronously execute on the main actor (verified by process sample 2026-07-13). Declare such builders `nonisolated` or keep them on non-View types. `DocumentProgressRelay` forwards roughly 256 byte-delta-gated updates per log; per-frame forwarding stalled the main thread (beachball) for entire scans of multi-megabyte logs. Never access `DecodedLog.headerInfo`/`seriesCatalog` (computed, header reparse per access) inside per-row loops; hoist into a per-request context (see `BlackboxAnalysisWorkspace.RowContext`).

Fix: render from the FULL orientation. `AnalysisAttitudeEstimator.orientation` (new `AnalysisAttitudeOrientation` with forward/right/up body axes in the world frame x=fwd,y=right,z=up) is derived from gravity+heading (heading Gram-Schmidt orthogonalized, right = gravity x heading). `AnalysisCraftSample` now carries `orientation` instead of `upDirection`. `CraftRenderModel` stores forward/right/up in render space (x=right,y=nose,z=up); the app maps the estimator world frame to render by swapping each vector's x,y. `CraftProjection` is built from the three axes and removes only the world heading (rotation about vertical) so the nose stays up-screen like a chase camera, then feeds the fixed-elevation camera; pure roll/pitch produce mathematically identical coefficients to the old up-only projection, so tilt behavior is unchanged, but the in-plane twist through inverted is now correct (a roll stays a roll, no Z spin). Verified on the log: forward axis stays fixed through the roll while right/up rotate ~265 deg about it, orthonormal. The estimator `upDirection`/`angles` and `attitudeTimeline` stay for tests.
- Craft yaw-removal removed entirely (2026-07-20, final decision): removing the heading dynamically is fundamentally singular no matter how it is computed, and both variants were tried and rejected. Nose-azimuth removal (`atan2(forward.x, forward.y)`) is right for rolls (forward is the roll axis, so its azimuth is constant) but whips at the 27 s pitch flip when the nose crosses vertical (horizontal projection -> 0, azimuth jumps ~180 deg). Swing-twist removal about the vertical is right for pitch flips but winds during the 10 s roll because that roll is about a tilted (non-horizontal) forward axis, so the twist-about-vertical changes -> the Z spin returned. Measured both on btfl_007.bbl to confirm the mirror-image failures.
