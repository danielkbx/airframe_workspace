# Tooling Knowledge Base

This file captures practical lessons for using local tools efficiently and correctly in this workspace.

Agents must update this file proactively when they learn a better command, workflow, log strategy, simulator flow, parser oracle, test approach, or when they had to correct a tool usage mistake.

## General Rules

- Prefer CLI-first workflows.
- Use Xcode itself only for special cases where CLI tools are insufficient.
- Use `rg` / `rg --files` for fast codebase search.
- Keep planning and durable context in `.agents/`, not in the upstream clone.
- Do not use GSD workflows, GSD skills, `.planning/`, or GSD-generated roadmap/backlog structures in this workspace. Use the project's own `.agents/` files instead.
- Do not modify `blackbox-log-viewer/` unless the user explicitly asks.
- Do not introduce external dependencies without the user's explicit approval.
- Keep all project artifacts in English, including code, variables, documentation, agent files, comments, and commit text, even if the chat is in German.
- Before writing or reviewing any file under `Airframe/`, official docs, commit messages, PR titles, or PR descriptions, make sure there is no AI/agent/Codex/ChatGPT attribution. Keep that context only in `.agents/`.
- Human-visible text should be clear developer prose, slightly casual when appropriate, and never stiff generated boilerplate.
- The workspace root is now its own private Git repo. Check `git rev-parse --show-toplevel` before commit-sensitive work because the root repo, `Airframe/`, `blackbox-log-viewer/`, and `betaflight/` are separate Git repositories linked by submodules.
- Use `./scripts/ws-status.sh` from the root to inspect root and submodule working trees together.
- Use `./scripts/ws-pull.sh` from the root for routine synchronization. It fast-forwards the read-only upstream reference submodules and leaves any pointer changes visible for an explicit root commit. It intentionally does not initialize nested upstream submodules such as Betaflight's firmware dependency modules.
- Use `./scripts/ws-commit-root.sh "message"` for normal root workspace commits when submodule worktrees are clean. It stages only root-owned paths, agent entry points (`AGENTS.md`, `CLAUDE.md`), and submodule pointers.

## Swift and Xcode

- Use separate iOS and macOS entitlement files when one Universal target needs macOS App Sandbox plus iCloud Key-Value Storage. Set normal `CODE_SIGN_ENTITLEMENTS` to the shared iOS file and use `CODE_SIGN_ENTITLEMENTS[sdk=macosx*]` for macOS.
- Verify a built app's effective signing with `codesign -d --entitlements :- /path/to/Airframe.app`; project settings alone do not prove the final signed payload.
- Before iOS simulator tests, inspect available devices with `xcrun simctl list devices available`; simulator names are machine-local and can drift from old commands.

- Use Swift 5.9 language mode unless the user explicitly approves a change.
- For Swift Packages, use `swift build` and `swift test` directly where possible.
- Use `xcodebuild` for package or app scheme builds/tests when required.
- Use `xcodeproj` for Xcode project file inspection or edits when appropriate.
- Use `xcrun simctl` for simulator lifecycle, app install/launch, screenshots, and log collection.
- For simulator-related `xcodebuild`, do not pipe output to `grep`.
- Instead, write useful log files, wait for `xcodebuild` to finish, and inspect the logs afterwards.
- Prefer modern Swift Testing for package tests.

## Current Tool Corrections Learned

- In zsh shell wrappers, do not assign to `status`; it is a read-only special parameter. Use a neutral name such as `result` for a command exit code.
- Swift Testing parameterized test methods cannot use `private` or `fileprivate` argument helper types unless the test method access is restricted the same way. Prefer internal test case helper structs for parameterized tests to keep test discovery straightforward.
- In this workspace, `swift test` with Swift Testing prints an XCTest compatibility summary with `Executed 0 tests` before the real Swift Testing run. Check the later Swift Testing summary; the initial XCTest line alone is not the result.
- The Swift project directory is `Airframe/`. Set the command working directory directly to package directories such as `Airframe/Packages/BlackboxCore` before running `swift test` or `swift build`.
- After renaming or moving a SwiftPM package directory, remove the package `.build/` directories before testing. Swift/Clang module cache files can retain the old absolute path and fail with `SwiftShims` module-cache path mismatches.
- When using `rg` patterns that contain Markdown backticks, wrap the whole pattern in single quotes. Unescaped backticks inside a double-quoted zsh argument trigger command substitution.
- SwiftPM `.process("Fixtures")` test resources may be available at the resource bundle root rather than under the original subdirectory. Fixture loaders should first try `subdirectory: "Fixtures"` and then fall back to a root lookup.
- For Swift ArgumentParser `ParsableArguments` used in `@OptionGroup`, avoid custom initializers on the parsable wrapper type unless they are proven compatible with completion generation. Keep the parsable options type internal and map it to a separate plain model type for service tests; this keeps `--generate-completion-script` / wrapper aliases such as `--completion bash` working.
- When a command type name would shadow Foundation types, such as a `data` subcommand, give the Swift type a different name like `DataCommand` and set `CommandConfiguration(commandName: "data")`.
- Airframe CLI `.bbfixture` files under test resources are a fixture DSL, not direct CLI input. They must be materialized by `CLIFixtureLoader` in tests; running `.build/debug/airframe info Tests/.../info-rich.bbfixture` directly fails with schema/header errors because the file is not a raw Blackbox log.
- Use root `Flightlogs/` for copied full private logs used by manual smoke/performance checks. Keep package test fixtures minimal and prepared under the package test target.
- Do not run `swift test` and `swift build` in parallel for the same package directory. SwiftPM serializes access to the shared `.build` directory with a lock, so the second command just waits. Run them sequentially when measuring package verification time or reading clean output.
- Do not run two `xcodebuild` commands in parallel against the same project and DerivedData location. Xcode can fail with `XCBuildData/build.db: database is locked`. Run them sequentially or give each command an isolated `-derivedDataPath`.
- When a SwiftPM package consumes a local path dependency and newly added source files in that dependency are not visible from the consuming package, remove the consuming package's `.build/` directory and rerun `swift test`. This occurred for `AirframeCLI` after adding new `BlackboxReader` source files.
- For local-only full-log smoke checks without adding repo files, create a temporary SwiftPM executable under `/tmp` that depends on the local package by absolute path, then run it against `Flightlogs/small/*`. This avoids putting private logs or smoke harnesses into package tests while still validating real log iteration.
- SwiftPM packages without explicit platform declarations may need `@available(macOS 10.15, iOS 13, tvOS 13, watchOS 6, *)` on public APIs that use `withTaskGroup`, even when async package tests run on a modern host.
- This local SwiftPM version does not support `swift package init --package-path`; create the target directory first and run `swift package init --type executable` with that directory as the working directory.
- When creating a temporary SwiftPM executable, make sure the shell actually `cd`s into the target directory before `swift package init`; setting a not-yet-existing directory as the command working directory fails, and running from `/tmp` initializes the package in `/tmp`.
- A temporary executable package that calls `BlackboxReader.summaries(in:)` needs the `@available(macOS 10.15, *)` annotation on the `@main` type, not only on `main()`.

- For `git diff` on Airframe project files, run the command from `/Users/daniel/Projekte/Blackbox/Airframe` with repo-relative paths such as `Packages/BlackboxReader/...`. Running `git diff -- Airframe/Packages/...` from the workspace root is outside the Airframe repository and fails like a no-index comparison.

- SwiftUI `View` is a MainActor-annotated protocol, so EVERY member of a conforming type (including statics in extensions) is MainActor-isolated. A `Task.detached` closure that synchronously calls such a member executes it on the main actor in the real app even though unit tests replicating the pattern can still run it off-main (Swift 5.9 minimal checking behaves differently across contexts). Never put heavy worker functions on a View type; declare them `nonisolated` or move them to a non-View namespace. Verify with a process sample (`sample <pid>`), not only with tests: the tell-tale stack is `thunk for @escaping @isolated(any) ... -> closure in <View>.method` under `com.apple.main-thread`.
- Launching the macOS app with a document: use `open -a <app-path> <file>`. Also note: LaunchServices routes documents to an already-running instance with the same bundle ID (for example the user's Xcode-launched build), not necessarily the app at the given path. Before measuring, check `ps aux | rg Airframe.app/Contents/MacOS` for competing instances, do not kill or launch instances while the user is actively working, and sample the PID that actually owns the document.

Never access them inside per-row/per-frame loops — hoist them (and derived lookups such as sorted field names or header-based ranges) into a per-request context.
